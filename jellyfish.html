Objective : Identify a change to production code, run only the relevant tests so as to optimize the execution time, efficiency of tests and enhance productivity.

Problem Statement : Today, It's a very common challenge for the agile teams to determine the relevant subset of tests to be executed for certifying a change in production code. Functional test suite keeps growing with every sprint. code bases like control center we have close to 15,000+ tests and with shorter sprint cycle of 1/2 weeks, it's quite challenging to identify the right set of tests to be executed for a given source code change. General practise is since "Tests are Automated, Run Them ALL !!" But, If tests ran infinitely quickly, we would run all the tests all the time, but they are not, so we need to balance cost vs value when running them.

"Test Impact Analysis (TIA) is a modern way of speeding up the test automation phase of a build. It works by analyzing the call-graph of the source code to work out which tests should be run after a change to production code." The Idea is to build an Intelligence in the software testing framework itself (either its a Junit or functional, Integration automation tests) to determine the exact set of tests that are really needed to certify a code-change. To build this solution we need to track the dependency between source classes\methods and mapping of tests with the source code.
